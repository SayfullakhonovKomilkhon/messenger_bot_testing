# Тестирование бота мессенджера

Два компонента:

1. **`test_client.py`** — локально дергает все методы Bot API (`/me`, диалоги, сообщения, вебхук, опционально отправка).
2. **`webhook_server.py`** + **`reply_logic.py`** — сервис для **Railway**: POST `/webhook` → ответы по простым правилам (привет, «как дела», запрос номера, распознавание RU-телефона в тексте). Проверяет **HMAC-подпись** входящего запроса, ретраит `sendMessage` один раз на HTTP 429 (per-bot rate-limit на бэкенде). При `ECHO_REPLY=0` ответы отключены.

**Не как в Telegram:** API мессенджера не поддерживает inline-кнопки и «Поделиться контактом» — только текст (и файлы). Номер пользователь присылает сообщением.

## Сценарий, который реализован

Пользователь пишет боту — бот отвечает по правилам из `reply_logic.build_reply`:

| Вход | Реакция |
|------|---------|
| «Привет» / «Hi» / «Здравствуйте» / «Добрый день» и т. п. | Приветствует и просит номер. Диалог переходит в состояние «жду номер» (TTL 15 мин). |
| Любое сообщение в состоянии «жду номер», содержащее валидный RU-номер (`+7…`, `8…`, `10 цифр`) | Подтверждает: «Спасибо! Номер принят: +79001234567» и сбрасывает состояние. |
| Сообщение в том же состоянии, но без номера | «Похоже, это не номер. Пришлите цифрами…» |
| «Номер», «телефон», «позвонить» вне состояния | Просит прислать номер сообщением. |
| «Как дела» | Короткий ответ + намёк, что может записать телефон. |
| Всё остальное | Эхо с подсказкой доступных команд. |

Состояние «жду номер» хранится в памяти процесса (dict + TTL). При рестарте сервиса — сбрасывается; это допустимо для тестового бота. Для прод-бота перенесите в Redis.

## Безопасность

- **Не коммитьте** файл `.env` и **не вставляйте токен в код**. Токен из чата/скринов лучше **перевыпустить** в приложении (регенерация токена бота), если его кто-то мог увидеть.
- На Railway задавайте секреты в **Variables**, не в репозитории.

## Локально: проверка API

```bash
cd tg-bot-testing
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
```

В `.env` укажите:

- `MESSENGER_API_URL` — **публичный URL bot-gateway** для вызовов `/api/v1/bot/*` (без `/api/v1` в конце; скрипт сам добавит путь). Пример локально: `http://127.0.0.1:3001`. Ядро на `:3000` эти маршруты больше не отдаёт; CRUD ботов (`/api/v1/bots`) по-прежнему на ядре, если понадобится отдельный скрипт — задайте другой базовый URL вручную.
- `BOT_TOKEN` — токен бота.

Опционально:

- `TARGET_USER_ID` — UUID **человека** (не бота), чтобы проверить `startConversation`.
- `CONVERSATION_ID` — UUID чата с ботом, если список диалогов пуст с точки зрения API.
- `WEBHOOK_PUBLIC_URL` — после деплоя вебхука, например `https://xxx.up.railway.app/webhook`.
- `TEST_SEND_MESSAGE=1` — отправить тестовую строку в `CONVERSATION_ID` (осторожно: реальное сообщение в чат).

Запуск:

```bash
python test_client.py
```

В выводе будет `OK` / `FAIL` по каждому вызову.

## Railway: вебхук + эхо

1. Создайте проект из этой папки (`tg-bot-testing` как root).
2. **Variables:**
   - `MESSENGER_API_URL`
   - `BOT_TOKEN`
   - `ECHO_REPLY` = `1` (или `0`, чтобы только логировать входящие POST без ответа)
   - `VERIFY_SIGNATURE` = `1` (обязательно в проде — бэкенд всегда подписывает)
   - `SIGNATURE_MAX_SKEW` = `300` (сколько секунд считать подпись свежей)
3. После деплоя скопируйте публичный URL сервиса, например `https://your-service.up.railway.app`.
4. Установите вебхук (один из способов):
   - в `.env` локально: `WEBHOOK_PUBLIC_URL=https://your-service.up.railway.app/webhook` и снова `python test_client.py`,  
   - или в приложении мессенджера укажите тот же URL вебхука для бота.
5. Напишите боту из приложения — на Railway в логах должен появиться payload; при `ECHO_REPLY=1` бот ответит по правилам из `reply_logic.build_reply`.

Проверка живости сервиса: `GET /health`.

## Безопасность webhook'а

Каждый POST на `/webhook` бэкенд подписывает: `X-Bot-Signature: t=<unix>,v1=<hex(HMAC_SHA256(BOT_TOKEN, "<t>.<raw_body>"))>` (+ `X-Bot-Id`). `webhook_server.py` при `VERIFY_SIGNATURE=1`:

1. Читает **сырые байты** тела запроса (до парсинга JSON) — иначе HMAC не сойдётся.
2. Проверяет, что `t` не старше `SIGNATURE_MAX_SKEW` секунд — защита от replay-атак.
3. Сверяет hex-дайджест через `hmac.compare_digest` (constant-time).
4. На любой сбой — отдаёт `401`.

Отключайте `VERIFY_SIGNATURE` **только** локально и только если запускаете против бэкенда без подписей. В проде это равносильно открытой дыре — любой может прислать фейковый webhook и вызвать лишние `sendMessage`.
